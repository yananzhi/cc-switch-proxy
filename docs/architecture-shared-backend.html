Claude Code Proxy — 多窗口共享架构与隔离缺陷

当前架构中，所有 VS Code 窗口共享同一个 Proxy Backend 进程和同一份 ~/.claude/settings.json，导致「Workspace Local LLM Config」并未真正实现 per-workspace 隔离。

1完整架构：多窗口共享单例

两个 VS Code 窗口打开不同 Workspace，各自有独立的 Local Config，但共享同一个 Proxy Backend 进程和同一个 Upstream 出口。官方 Claude Code 插件 UI 无法注入 CLAUDE_CONFIG_DIR，只能读全局 ~/.claude/settings.json。

2两层隔离缺陷详解

「Workspace Local LLM Config」看似每个 Workspace 有独立配置，实际上有两层隔离没有做到。

3为什么现在"能用"——恰好没串到不同后端

当前 Global 和 Local 配置指向同一个 Upstream 后端（相同 baseUrl/token），只是 model ID 不同。所以 last-write-wins 只改变 model，不会让请求打到错误的后端。

4隔离能力总结

维度当前状态是否真正隔离原因
配置文件目录
CLAUDE_CONFIG_DIR		 每个 Workspace 有独立的 .claude_proxy/ ✅ 隔离 Launcher 启动 CLI 时注入 CLAUDE_CONFIG_DIR，CLI 读 workspace 本地的 settings.json
对话历史 / 会话状态 每个 Workspace 独立 ✅ 隔离 历史和会话存在 CLAUDE_CONFIG_DIR 下，天然隔离
官方插件 UI 配置 所有窗口共享 ~/.claude/settings.json ❌ 不隔离 官方插件不支持 CLAUDE_CONFIG_DIR，内部 CLI binary 始终读全局 settings.json；切换 Global Config = 改写全局文件 + Reload Window，所有窗口同时受影响
Proxy Upstream
（运行时转发目标）		 全局单例，只有一个出口 ❌ 不隔离 Proxy 进程靠端口 bind 保证全局单例；Upstream 存在 proxy-config.json，不在任何 CLAUDE_CONFIG_DIR 里；setUpstream() last-write-wins
Proxy 重试策略 全局共享 ❌ 不隔离 重试参数（maxAttempts、backoff）是 Proxy 进程全局配置，无法 per-workspace 定制
核心结论：「Workspace Local LLM Config」隔离的只是配置文件目录（settings.json、对话历史、会话状态），没有隔离运行时行为（Proxy Upstream、重试策略）。当前能用是因为所有配置恰好指向同一个 Upstream 后端（baseUrl/token 相同），只是 model ID 不同——last-write-wins 只改变 model，不会让请求打到错误的后端。这不是真正的隔离，是巧合。
真正隔离需要：
1. 官方插件 UI 隔离：需要官方 Claude Code 插件支持 CLAUDE_CONFIG_DIR 或类似机制，让不同窗口的插件 UI 读不同的配置文件。目前无法从外部注入。
2. Proxy Upstream 隔离：需要每个 Workspace 有独立的 Proxy 实例（独立端口、独立 Upstream 状态），参见 docs/pitfall-proxy-shared-upstream.md 中的方案。
