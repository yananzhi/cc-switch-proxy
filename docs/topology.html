Claude Code Switch + LLM 代理 — 网络拓扑

本插件的本质：通过改写 ~/.claude/settings.json 里的 env.ANTHROPIC_BASE_URL，决定 Claude Code「连到哪」——直连真实上游，还是先走本地代理再转给真实上游。

数据流（HTTP 请求/响应） 改写配置 / 注入上游（控制面） 心跳 / 生命周期 重试（指数退避） 不参与 / 断开

1代理模式：「通过代理连接」配置激活后

一条 mode=proxy 的配置被点激活时，扩展先把代理跑起来、把真实上游注入代理，再把 settings.json 的 baseUrl 改写成 http://127.0.0.1:<port>。重试发生在「代理 → 上游」这一段。

重试机制在哪里？ 在「代理 → LLM API」这一跳（图示 ⑤）。代理只重试 Claude Code 处理不了的——HTTP 503 + body error.code === 10310（讯飞 system busy）。为什么是这一条：查 Claude Code 源码（services/api/withRetry.ts），它关掉了 SDK 自带重试、自己手写重试，会把 503 当普通 5xx 盲目重试 10 次，但不识别 10310 这个业务码——代理的针对重试（20 次 + 16s 上限）对讯飞 system busy 更有效。其余全部透传交给 Claude Code：429/500/502/504（CC 当 5xx/限流重试）、网络错误/超时/断连/流中断（CC 当 APIConnectionError 重试，代理无法原样转 socket 错误，合成 502 回客户端，CC 仍当 5xx 重试，语义等价）。退避 backoffForMs = min(backoffSec*2^(n-1), 16s)，最多 20 次。
组件角色关键事实
VS Code宿主进程扩展和代理都跑在它的扩展宿主进程内；extensionKind: workspace，WSL 里代理与 Claude Code 同 localhost
Claude Code 插件 (cc-switch)编排者 / 控制面不碰数据流本身；只做三件事：拉起代理、注入真实上游、改写 settings.json 的 ANTHROPIC_BASE_URL
代理 (proxy/server.js)数据面 + 重试监听 127.0.0.1:11434；收到请求转发给真实上游；对 503/10310 等做指数退避重试；其余透传
LLM API 服务真实上游真实 baseUrl + token 由插件从配置 content.env 解出后注入代理；Claude Code 看不到真实地址（只看到 127.0.0.1）

2直连模式：「直连」配置激活后

默认模式。一条 mode=direct 的配置被激活时，扩展只是把配置 content 原样写进 settings.json，代理完全不参与——和原 cc-switch 行为一致。

为什么默认是直连？ 直连不引入任何额外跳数、不动网络路径，等于「不用代理」——只有切到 mode=proxy 的配置时代理才参与。这保证了「我只想切配置、不想要代理」的场景跟原 cc-switch 完全一致，零风险。

3Claude Code 本身根据什么、向哪里连？

不管哪种模式，Claude Code 运行时只认一个东西：~/.claude/settings.json 里的 env。本插件能「改连接方式」的全部原理，就在下面这张表里。

4实现要点：用代理 vs 不用代理，差在哪？

维度直连（mode=direct）代理（mode=proxy）
扩展激活时做的事 仅 writeSettings(content)，把配置原样写进 settings.json ① proxyHost.ensureRunning() 拉起代理
② setUpstream(真实baseUrl/token/model) 注入
③ synthesizeProxySettings() 把 BASE_URL 改成 127.0.0.1:port 后写入 settings.json
settings.json 里 BASE_URL 的值 真实上游 URL（配置 content.env 原值） http://127.0.0.1:11434（被改写）
Claude Code 看到的连接目标 真实上游 本地代理（看不到真实地址）
真实 token 存在哪 settings.json 里（Claude Code 直接读） 代理内存里（注入进去）；settings.json 里仍是真实 token，代理转发时用
数据流路径 Claude Code → LLM API（一跳） Claude Code → 代理 → LLM API（两跳）
503 / code 10310 怎么办 Claude Code 处理不了 → 报错给用户 代理拦截，指数退避重试 ≤20 次，成功后才回 Claude Code
其他状态码 + 网络错误
（429/500/502/504、超时/断连/流中断）		 两种模式都一样：Claude Code 自己会重试这些（5xx→重试、APIConnectionError→重试，各 10 次），代理不插手。代理模式下代理原样透传或合成 502 回客户端（retryOnStatus 为空），让 CC 当 5xx 自己重试。
代理是否参与 完全不参与（可能常驻但不注入上游） 必须参与：监听 + 注入上游 + 拦截重试
可观测性 无（Claude Code 自己的日志） Web 控制台看重试 trace + 详细日志（http://127.0.0.1:port/）
一句话总结： 本插件不改 Claude Code 本身的代码，只改它的「连接配置」——直连时把真实地址写进 settings.json；代理时把地址改成本地 127.0.0.1，让 Claude Code 以为在连上游、实际连的是代理，由代理转给真实上游并兜底重试。同一个开关（settings.json 的 env.ANTHROPIC_BASE_URL），两个值，两种连接方式。
